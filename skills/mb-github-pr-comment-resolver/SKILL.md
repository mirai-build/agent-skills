# ミライビルド GitHub PRコメントリゾルバー

## 概要

GitHub PR URL を起点に未解決の review thread を棚卸しし、項番つきで重大度、指摘内容、分析結果、推奨対応方針をまとめる。
ユーザーが対応方針を指定した後は、その方針に従って修正、コミット、返信、resolve まで完了させる。

## 使う場面

- GitHub PR の未解決コメントをまとめて分析し、どれを修正すべきか判断したい。
- レビューコメントに対する修正、説明返信、resolve を一括で進めたい。
- ユーザーが「全てその通りに」「項番 1 と 3 は修正、2 はコメントのみ」など、分析結果に対する対応方針を示した。
- 対象は GitHub の resolve 可能な PR review thread とする。トップレベルの PR Conversation コメントは resolve できないため、必要なら返信対象として別枠で扱う。

## 利用ツール

- GitHub App connector を優先して使う。PR 情報、review thread、コメント返信、resolve は GitHub App のツールで行う。
- GitHub App ツールがまだ見えていない場合は `tool_search` で GitHub 関連ツールを探す。
- `gh` CLI は、GitHub App connector だけでは取得できない補助情報や CI 状態確認が必要な場合の補助に限る。
- ローカル修正が必要な場合は、対象 PR の head branch に対応するリポジトリを確認し、通常のコード編集、テスト、commit、push の流れで進める。

## 実行原則

- いきなり修正や resolve を始めない。最初の応答では必ず未解決 thread の分析結果を提示し、ユーザーの対応判断を待つ。
- ユーザーが「全てその通りに」と指示した場合だけ、提示した推奨対応方針を承認済みとして扱う。
- コメントは必ず日本語で書く。PR タイトル、PR コメント、review thread への返信、commit message も日本語にする。
- resolve は、対応内容の返信を投稿した後に行う。無言 resolve はしない。
- 修正ありの項目は、修正内容が PR 上で確認できる状態にしてから返信と resolve を行う。通常は commit 後に PR head branch へ push する。
- 修正なしの項目も、対応しない理由や確認結果を返信してから resolve する。
- 既存の未コミット変更がある場合は、ユーザーや他作業の変更を巻き戻さず、対象差分との関係を確認してから進める。
- 判断が割れる指摘は、勝手に対応不要にせず、分析結果に不確実性を明記してユーザー判断に回す。
- すべての対象 thread について、最後に「返信済み」「resolve 済み」「未完了理由」のいずれかを報告する。

## 基本フロー

### 1. PR URL と対象情報を確認する

1. ユーザーの依頼から GitHub PR URL を抽出する。URL がない場合は、対象 PR URL を確認してから進める。
2. URL から `owner/repo` と PR 番号を特定する。
3. GitHub App connector で PR metadata、head branch、base branch、差分、review thread 一覧を取得する。
4. `isResolved` 相当の状態を見て、未解決 review thread だけを対象にする。
5. すでに resolved の thread、bot のノイズ、トップレベル Conversation コメントは、必要に応じて「対象外」として件数だけ報告する。

### 2. 未解決 thread を分析する

各 thread について次を確認する。

- コメント本文、投稿者、対象ファイル、対象行、thread ID、古い差分へのコメントかどうか。
- PR の現在差分と、必要なら対象ファイルの現状コード。
- 指摘が実バグ、仕様確認、設計改善、可読性、テスト不足、単なる質問のどれに近いか。
- 修正が必要な場合の修正範囲、影響範囲、検証方法。
- 修正しない場合に説明できる根拠。

重大度は次の基準で付ける。

- `P0`: merge すると重大障害、データ破壊、セキュリティ事故につながる可能性が高い。
- `P1`: 機能不具合、仕様逸脱、テストで拾うべき回帰など、merge 前に直すべき問題。
- `P2`: 保守性、設計、可読性、テスト補強など、直す価値が高いが重大障害ではない問題。
- `P3`: 質問、表現、任意改善、対応不要に近い軽微な指摘。

対応方針は次のいずれかを基本にする。

- `修正`: コードやドキュメントを変更し、検証、commit、push、返信、resolve まで行う。
- `コメントのみ`: 現状で妥当と判断できるため、根拠を返信して resolve する。
- `確認待ち`: 仕様判断やレビュワー意図の確認が必要で、ユーザーに追加判断を求める。
- `対象外`: すでに解消済み、古い差分、resolve 不能なコメントなど、今回の処理対象から外す。

### 3. ステップ1の分析結果を提示する

最初の出力では、修正や resolve を行わず、表は使わずに見出しと小見出しでユーザーに判断を仰ぐ。
全体見出し、対象ごとの項番見出し、判断依頼の見出しを分け、各項目の中で重大度、場所、指摘内容、対応要否、分析結果、推奨対応方針を示す。

```text
## 未解決コメント分析

### 項番 1: 〜〜の指摘

#### 重大度
P1

#### 場所
path/to/file.ts:42

#### 指摘内容
〜〜

#### 対応要否
要対応

#### 分析結果
〜〜

#### 推奨対応方針
修正: 〜〜

## 判断をお願いします

「全てその通りに」と言われたら、上記の推奨対応方針どおりに進めます。
個別指定する場合は「項番 1 は修正、2 はコメントのみ、3 は保留」のように返して下さい。
```

分析結果の後に次を添える。

- 「全てその通りに」と言われたら、推奨対応方針どおりに進めること。
- 個別指定する場合は「項番 1 は修正、2 はコメントのみ、3 は保留」のように返してほしいこと。
- `確認待ち` の項目がある場合は、判断に必要な選択肢。

### 4. ユーザーの対応判断を解釈する

ユーザーから次のような指示を受けたら、対応フェーズへ進む。

- 「全てその通りに」: ステップ1で提示した推奨対応方針を採用する。
- 「1,3 は修正、2 はコメントのみ」: 指定項番だけを指定方針で処理する。
- 「2 は対応しないで resolve」: 対応不要理由を返信して resolve する。
- 「3 は保留」: その項目は処理せず、最終報告で未完了として残す。

指示が曖昧で、修正するかコメントのみか判断できない項目は、作業に入る前に短く確認する。

### 5. 修正あり項目に対応する

1. ローカルリポジトリが対象 PR の head branch と一致しているか確認する。
2. 作業ツリーが dirty の場合は、既存変更と今回対応の関係を確認する。無関係な変更を戻さない。
3. 指定方針に沿って最小範囲で修正する。
4. 影響範囲に応じて lint、型検査、テスト、ビルドを実行する。
5. 修正内容を日本語 commit message で commit する。複数項目が密接に関連する場合は 1 commit にまとめてよい。
6. PR 上で確認できるよう、可能なら head branch へ push する。
7. 各 thread に、対応内容、commit hash、検証結果を日本語で返信する。
8. 返信後に対象 thread を resolve する。

返信例:

```text
対応しました。

- 変更内容: 〜〜
- コミット: abc1234
- 検証: pnpm test を実行し成功

この thread は resolve します。
```

### 6. コメントのみ項目に対応する

1. 修正しない根拠を、コードや仕様、既存挙動に基づいて短く整理する。
2. 対象 thread に日本語で返信する。
3. 返信後に対象 thread を resolve する。

返信例:

```text
確認しました。
この箇所は 〜〜 のため、現状の実装で意図どおりです。
追加修正は行わず、この thread は resolve します。
```

### 7. 最終報告をまとめる

最終報告は次の順で簡潔にまとめる。

1. 対象 PR
2. 対応した項番と方針
3. 作成した commit と push の有無
4. 実行した検証
5. 返信、resolve の完了状況
6. 保留または未完了項目と理由

## 注意点

- GitHub の review thread ID は resolve 操作に必要なので、分析時に内部で控える。ユーザー向けには各小見出しの項番を主キーとして出す。
- review thread へ返信するには、thread 内の先頭 inline review comment ID が必要になるため、thread ID と一緒に控える。
- outdated だが未解決の thread は、現行コードで解消済みかを確認し、解消済みなら `コメントのみ` として返信して resolve する。
- レビュワーへの質問が必要な thread は、勝手に resolve せず、ユーザーが resolve 方針を明示した場合だけ返信して resolve する。
- GitHub App connector の resolve 操作が失敗した場合は、対象 thread、投稿した返信、失敗理由を報告し、再試行可能な状態で止める。
