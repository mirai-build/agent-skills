# ヒアリングガイド

この Skill では、足りない情報を質問しながら DDD の設計書を深めていく。
この資料は、フェーズごとにどんな観点でユーザーへ情報提供をお願いするとよいかをまとめたガイドである。

## 共通ルール

- まず既存資料を確認し、読めば分かることを質問しない。
- `docs/designs` の成熟度を確認し、今必要な次のフェーズだけに質問を絞る。
- 上流文書を変えるときは、既存の下流文書にどんな影響が出るかも先に確認する。
- 1 回の質問は 1〜3 テーマに絞る。
- 抽象的な質問だけを投げず、回答例や選択肢を添える。
- 回答が難しそうなときは、仮説を置いて「近いかどうか」を確認する。
- ユーザーの負担が大きい場合は、今のフェーズに必要な最低限の確認から進める。

## 最初に成熟度を見極めるとき

- `domain_model` はどこまであるか。少なくとも `01_system_purpose` `02_actors` `03_usecases` `04_journeys` `05_bounded_contexts` が順に読めるか。
- `contexts` はどこまであるか。どの bounded context が詳細化済みで、どこが未着手か。
- `implementation` へ進めそうな context はあるか。それともまだ domain / context 設計を先に固めるべきか。
- 今回はどの 1 ステップだけ進めるのが最も自然か。

## 最初にお願いするとよい資料

- 企画書、要件定義、業務フロー、既存設計書
- 画面一覧、ワイヤーフレーム、API 仕様、DB スキーマ
- 運用手順、障害事例、監査要件、法規制
- KPI、成功指標、チーム体制、既存の用語集
- 既存のアクター一覧、ユースケース一覧、権限表、状態遷移図

## システムの目的整理で聞くこと

### 概要

- このシステムは誰のための何か。
- 新規開発か、既存業務の置き換えか、既存プロダクトの拡張か。
- いま 1 文で説明するならどう言うか。

### 解決する問題

- 現在どんな業務上の困りごとが起きているか。
- その問題を放置すると何が困るか。
- 問題が最も強く表れる場面や頻度はどこか。

### 対象ユーザー

- 主に使う人は誰か。現場担当者、管理者、外部顧客のどれに近いか。
- 利用者ごとに、成功と失敗をどう判断するか。
- 利用者以外に、承認者、監査担当、運用担当などの関係者はいるか。

### 目指す状態

- 導入後に何ができるようになっていれば成功か。
- 定量的に見たい指標はあるか。
- どこまで自動化、標準化、可視化したいか。

### 解決アプローチ

- どの業務判断をシステム化したいか。
- 人が残す判断と、システムへ委ねる判断をどう分けたいか。
- 外部システム、AI、手作業との役割分担はどうするか。

### プロダクトスコープ

- 最初のリリースで必ず含めたい範囲はどこまでか。
- あえて今回外すものは何か。
- 将来の拡張として想定している領域はあるか。

### 提供価値

- 誰に対して、どんな価値を提供できると考えているか。
- その価値は時間短縮、品質向上、売上、リスク低減のどれに近いか。
- 競合や既存運用と比べて、何が差別化ポイントか。

### 設計原則

- 業務で絶対に壊してはいけないルールは何か。
- 説明責任、監査性、追跡性、即時性のうち、特に重視するものはどれか。
- 今後の変更容易性で優先したい観点は何か。

## アクター定義を整理するとき

### 質問例

- このシステムに関わる actor は誰か。利用者、管理者、運用担当、外部システムのどれか。
- その actor の主目的は何か。
- どのユースケースを主に開始するか、または結果を受け取るか。
- actor ごとに権限や見える情報の差はあるか。
- 人ではなく外部システムや定期実行も actor として扱うべきか。

## bounded context 分割を探るとき

### 軽量イベントストーミングの進め方

1. 主要なアクターを列挙する。
2. それぞれのアクターが、何をきっかけに何を行うかを時系列で並べる。
3. 行動の結果として何が起きたかを、過去形のイベントとして並べる。
4. そのイベントに影響するルール、例外、外部連携、承認ポイントを洗い出す。
5. 判断が難しい箇所、現場の痛みが強い箇所、差別化の核になる箇所をホットスポットとして残す。

### 質問例

- その業務は誰が始め、何が起きたら次の担当へ渡るか。
- 業務上「完了した」と判断する瞬間はいつか。
- 手戻り、差し戻し、キャンセルはどこで起こるか。
- 例外対応や属人化が起きやすいのはどこか。
- 他社や既存ツールと最も差がつく判断はどこか。

### 整理の補助

- コア業務
  - プロダクト価値の中心で、利用者に直接価値を届ける業務領域。
- 支援業務
  - コア業務を成立させるために必要だが、価値提供の中心ではない業務領域。
- サブドメイン
  - コア業務または支援業務の中で、さらに分けて考えた方が責務や会話が明確になる場合に切り出す単位。

整理に迷うときは、次の順番で確認する。

1. まずコア業務と支援業務を分ける。
2. そのうえで、各業務の中に明確な責務差や用語差があるならサブドメインへ分割する。
3. 特にコア業務は、責務やユビキタス言語が分かれるなら bounded context 候補まで具体化する。
4. 支援業務も、独立した責務や用語があるなら bounded context 候補として切り出す。
5. 無理に細分化せず、会話や設計判断に必要な粒度だけ切り出す。

コア業務かどうかを見極めるときは、次の観点で確認する。

- ここが弱いとプロダクト価値が大きく落ちるか。
- ここに独自ルールや高度な判断が集中しているか。
- 他社との差別化が最も出やすいのはどこか。

用語のぶれが大きい場合は、早い段階で用語集テンプレートを追加し、設計書の正本をそろえる。

## ユースケースを整理するとき

ユースケースは `domain_model/03_usecases/` 配下で扱い、`00_overview.md` を入口に `01_*.md` で分割管理する。
後続の `contexts` や `implementation` で参照される前提として、主要 actor、開始条件、完了条件、主な flow をそろえる。

### 質問例

- このシステムで利用者や運用者が実行する主要な操作は何か。
- 誰が、何をきっかけに、その操作を開始するか。
- 正常完了とは何を意味するか。
- 代表的な例外や差し戻しは何か。
- どの bounded context が主にそのユースケースを担うか。

## ユーザージャーニーを整理するとき

ユーザージャーニーは `domain_model/04_journeys/` 配下で扱い、`00_overview.md` を入口に `01_*.md` で分割管理する。
感情分析ではなく、複数 actor、複数画面、手作業、外部連携、複数 bounded context をまたぐ一連の流れを、関連ユースケースをつないだ簡易フローとして整理したいときに追加する。

### 質問例

- どのユーザーが、どんなきっかけでこの流れを開始するか。
- 開始から完了までに、どの画面、承認、通知、外部連携が登場するか。
- 各ステップで、どのユースケースが実行されるか。
- 途中で離脱、差し戻し、失敗が起きるのはどこか。
- このジャーニーの中で、どの bounded context がどの責務を担うか。

## コンテキスト配下の設計対象を掘るとき

### Value Object

- 同一性ではなく値で比較したい概念は何か。
- 生成時や更新時に必ず満たすべき条件は何か。
- 表示用の値と内部計算用の値を分ける必要はあるか。

### Entity

- 追跡すべき識別子は何か。
- どの状態遷移が業務上重要か。
- 履歴や監査が必要な変更はどれか。

### Aggregate

- どこまでを一貫性境界として同時に守る必要があるか。
- どの不変条件をトランザクション内で守りたいか。
- 同時更新の衝突が起きやすいのはどこか。

### Repository

- どの単位で読み書きしたいか。
- 参照系と更新系で必要な検索条件は何か。
- 履歴、スナップショット、論理削除などの永続化要件はあるか。

### Domain Service

- 1 つの Entity や Aggregate に閉じない業務判断はあるか。
- 複数概念をまたいで計算や判定を行う処理は何か。
- その処理は本当にドメイン知識か、アプリケーションの調停か。

### Application Service

- ユースケース開始時の入力は何か。
- 認可、トランザクション、外部連携の境界はどこか。
- 成功時と失敗時に呼び出し元へ何を返すか。

### Query Service

- 読み込み専用で切り出したい検索、一覧、集計、参照要件は何か。
- 画面や API へ返すために、どんな出力形が必要か。
- 更新責務や副作用が混ざっていないか。

### Interface

- 提供する入口は Web、API、バッチ、管理画面のどれか。
- どのユースケースに対するインターフェースか。
- 利用者に見せる入力、出力、エラーの責務境界をどう分けるか。

### Domain Event

- どの出来事を domain event として明示的に残すべきか。
- どの Aggregate や Service が発火元になるか。
- どの処理がその event を購読、連携、監査に利用するか。
- 冪等性、再送、順序保証で気を付ける点はあるか。

### External Integration

- どの外部 API、SaaS、他システムと連携するか。
- 連携方向は送信、受信、双方向のどれか。
- 認証、リトライ、タイムアウト、障害時の扱いをどうするか。
- どの context のどのユースケースがその連携に依存するか。

## 実装向け設計を掘るとき

実装着手前に API や DB まで設計書へ落としたい場合は、`implementation` 配下の文書も追加する。
ただし、対象 bounded context の `domain_model` と `contexts` が揃っていることを前提にする。
詳しい観点は [`implementation_design_guide.md`](./implementation_design_guide.md) を使い、ここでは会話の入口になる問いから始める。

### API

- この入口は誰が使い、何を完了したいのか。
- 同期 API、webhook、batch、worker のどれに近いか。
- request / response で最低限必要な項目は何か。
- 認可、冪等性、再実行方針で先に決めるべきことは何か。

### DB / 永続化

- どの Aggregate や read model を、どの保存単位で持つか。
- unique 制約や参照整合性で壊してはいけない条件は何か。
- 検索、並び替え、一覧取得のために必要な index は何か。
- 履歴保持、soft delete、監査列、保持期間で必要なものはあるか。

## コンテキスト設計で聞くこと

- 何をきっかけにユースケースが開始されるか。
- 前提条件、事前チェック、権限確認は何か。
- 正常系の最短フローはどうなるか。
- 例外系、失敗時の戻し、再実行方針はどうするか。
- 保存対象、更新順序、同時実行制御はどうするか。
- 外部 API や通知基盤とのやりとりは同期か非同期か。
- ログ、監査証跡、メトリクスとして何を残すべきか。
- テストで最初に担保したい観点は何か。

## 情報提供を促す言い回し例

- 「既存の業務フローや画面があれば、そのまま共有してもらえると用語のずれを減らせます。」
- 「判断が難しい箇所だけでも教えてもらえると、コア業務と支援業務の切り分けを進めやすいです。」
- 「正確な答えがなくても大丈夫なので、現場感に近いたたき台を教えて下さい。」
- 「用語が複数ある場合は、現場で最も自然に使う呼び方を優先して教えて下さい。」
