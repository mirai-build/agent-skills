# ヒアリングガイド

この Skill では、ユーザージャーニーを起点に画面構成を整理する。
ただし、前提となる `domain_model` が不足している場合は、この資料で深掘りせず `mb-ddd-architect` へ委譲する。

## 共通ルール

- まず既存の `docs/designs/domain_model` と `ux/` を確認し、読めば分かることを質問しない。
- `01_system_purpose` または `04_journeys` が不足しているときは、自分で補完せず、どの前提が足りないかを整理して返す。
- 1 回の質問は 1〜3 テーマに絞る。
- 抽象的な質問だけで終わらせず、候補、例、仮説案を添える。
- ヒアリング結果は、確定事項、仮置き、未確定事項に分けて整理する。

## 最初に確認すること

- この機能や画面は、どのユーザーが使うか。
- ユーザーが何を達成できれば成功か。
- 主にどの端末で使うか。PC、スマートフォン、タブレットのどれが主か。
- 既存の画面、ワイヤー、スプレッドシート、会話メモなどの材料があるか。
- どのジャーニーを今回の対象にするか。

## 前提不足のときに返すべきこと

- 足りない前提文書
  - `01_system_purpose/00_overview.md`
  - `04_journeys/00_overview.md`
- 足りないと何が決められないか
  - 主役ユーザーが決まらない
  - 画面の完了条件が決まらない
  - 遷移の起点と終点が決まらない
- `mb-ddd-architect` 側で最低限そろえてほしいこと
  - システムの目的
  - 主な actor
  - 主要ジャーニー

## `basic_layouts` を整理するとき

### ナビゲーション方針

- グローバルナビは何の単位で分けるか。
- ローカルナビは必要か。必要なら、どの画面群で出し分けるか。
- ロール差分は、非表示、閲覧専用、別メニューのどれで扱うか。

### 画面の大分類

- 現場業務画面、参照画面、管理画面、連携・同期画面のうち、今回どの分類があるか。
- 分類ごとに、共通した操作や表示の特徴はあるか。

### レイアウトの基本型

- 一覧画面、詳細画面、実行・確認画面、管理設定画面のうち、どの型が必要か。
- 1 つの基本型で足りるか。複数の基本構成パターンが必要か。

### 共通 UI 領域

- ヘッダー、サイドナビ、パンくず、フィルタバー、右サマリパネルのうち、どれを共通で持つか。
- 常時表示が必要な領域と、画面によって省く領域はどれか。
- 共通 UI 領域は、存在するものだけを基本構成へ書く前提にする。
- 本来あるべきところを意図的に外している領域がある場合だけ、その不採用理由を確認する。

## `screen_inventory` を整理するとき

### 画面候補の洗い出し

- このジャーニーは、どの画面から始まるか。
- 途中で必ず通る画面は何か。
- 一覧、詳細、入力、確認、完了、エラーのうち、どの種類の画面が必要か。
- 既存画面を再利用できるか。それとも専用画面が必要か。
- 同じ画面を複数のジャーニーで共有するか。

### 一覧に載せる内容

- 画面名を一言でどう説明するか。
- どのジャーニーに対応する画面か。
- どの基本構成を参照する画面か。
- 主要 actor は誰か。

## `navigation_map` を整理するとき

### 正常遷移

- どの画面からどの画面へ進むか。
- どの操作、またはどの条件で次の画面へ進むか。
- 完了時に戻る先はどこか。

### 例外遷移

- キャンセルしたとき、どこへ戻るか。
- エラー時にその場へ留まるか、別画面へ逃がすか。
- 権限不足や状態不整合で遷移できない場面はあるか。
- 離脱後に再開する導線は必要か。

## `screen_details` を整理するとき

### 画面の役割

- この画面でユーザーは何を確認するか。
- この画面でユーザーは何を判断するか。
- この画面でユーザーは何を操作するか。
- この画面では扱わないことは何か。

### 基本構成との関係

- どの基本構成パターンを参照する画面か。
- 基本構成で定義した共通 UI 領域のうち、この画面で使うものはどれか。
- この画面固有で補足すべき情報や操作は何か。

### 主要状態

- 通常状態では何が見えていればよいか。
- データがないとき、何を案内するか。
- 入力や業務エラーが起きたとき、どこで何を伝えるか。
- 権限が足りないとき、閲覧不可か、一部制限か、代替導線ありか。

## 答えが出ないときの進め方

- 仮説案を 1 つ提示し、「この理解で近いですか」と確認する。
- 画面名がまだ決まらない場合は、役割ベースの仮名を置き、正式名称は未確定として残す。
- 基本構成が決めづらい場合は、まず「ナビの単位」「共通 UI 領域」「レイアウトの基本型」だけを先に固定する。
