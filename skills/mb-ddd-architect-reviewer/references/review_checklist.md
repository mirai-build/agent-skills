# DDD 設計レビュー・チェックリスト

レビュー時は、次の順で確認する。
上流が揃っていない場合は、下流の細部より先に上流の欠落を指摘する。

## 1. 設計棚卸し

- `docs/design` と `docs/designs` のどちらが正本か分かるか。
- 対象範囲が「全体」「特定 bounded context」「実装直前レビュー」のどれか明示されているか。
- 要件、業務フロー、画面、API、DB、運用資料など、レビュー根拠がどこにあるか分かるか。
- overview 文書や入口ファイルがあり、詳細への導線が切れていないか。

### 赤信号

- 同じ主題の文書が複数あり、どれが最新か分からない。
- 図だけ、または表だけで、文章による責務説明がない。
- 特定 context の詳細はあるが、全体の目的や位置付けが見えない。

## 2. ドメイン理解とユビキタス言語

- システムの目的、対象ユーザー、主要 actor、主要 use case が読めるか。
- コア業務と支援業務の区別が説明できるか。
- 主要用語の日本語名とシステム上の呼び名が対応しているか。
- 同じ用語が文書、図、API、DB、コード候補で一貫しているか。

### 赤信号

- 画面名や API 名だけが先にあり、業務目的が分からない。
- 用語集がない、または用語の意味が文書ごとに変わる。
- 外部システムの語彙が、そのまま内部語彙になっている。

## 3. 戦略設計

- subdomain 分類があり、どこが core / supporting / generic か説明できるか。
- bounded context ごとに責務、主要概念、境界が説明されているか。
- context 間の関係が context map として読めるか。
- upstream / downstream や、外部境界を守る翻訳層の有無が分かるか。
- bounded context が技術レイヤではなく業務能力で切られているか。

### 赤信号

- context 名が `api` `batch` `db` のような技術用語になっている。
- 同じ actor / use case を複数 context が競合して担当している。
- context 間に依存はあるが、契約オーナーや翻訳責務が不明である。

## 4. 戦術設計

### Entity / Value Object

- Entity に識別子、主要状態、状態遷移があるか。
- Value Object に不変条件が閉じているか。
- 識別不要な概念を無理に Entity 化していないか。

### Aggregate

- aggregate root が明示されているか。
- 何を同時に守るべき不変条件とするか説明されているか。
- 複数 aggregate をまたぐ整合性をどう扱うか説明されているか。

### Service

- Domain Service が pure な業務判断に閉じているか。
- Application Service が orchestration と transaction 境界を担っているか。
- Query Service が読み取り専用に閉じているか。

### Repository / Event

- Repository が aggregate 境界と対応しているか。
- Domain Event の発火元、購読側、利用目的が分かるか。
- 非同期処理なら冪等性、再送、順序保証の扱いがあるか。

### 赤信号

- aggregate が巨大で、不変条件が整理されていない。
- Application Service が業務ルールを抱え込み、domain model が空洞化している。
- Repository が table 単位で分かれ、aggregate 単位で語れない。
- Domain Event と integration event が区別されていない。

## 5. 実装接続

- use case と API / UI / batch / worker の対応が追えるか。
- aggregate、repository、DB 設計の対応が追えるか。
- 認可、監査、履歴、エラー、差し戻し、再実行が漏れていないか。
- 外部連携契約や Webhook 契約があれば、payload、認証、失敗時処理まであるか。
- 用語差がある場合、どこで翻訳するか明示されているか。

### 赤信号

- API / DB 設計だけが先行し、上流の context 境界と矛盾している。
- 複数 aggregate を更新する処理があるのに、整合性方針がない。
- 外部連携のエラー時再実行や冪等性が未定義である。

## 6. レビュー結果のまとめ方

- まず `総評` で、対象範囲、正本、成熟度、総合所見、今回優先すべき論点を短くまとめる。
- `良い点` を 1〜3 件挙げ、残すべき判断を明示する。
- 修正方針が回答次第で変わる場合だけ `確認事項` を出し、その場合は回答が得られるまで `mb-ddd-architect` 向けの確定版修正指示を作らない。
- `指摘事項` は重大度順に並べ、最上位を最初に着手すべき修正とみなす。
- 各指摘に「根拠」「修正対象」「mb-ddd-architect への修正指示」「完了条件」「期待効果」を付ける。
