# ヒアリングガイド

この Skill では、足りない情報を質問しながら DDD の設計書を深めていく。
この資料は、フェーズごとにどんな観点でユーザーへ情報提供をお願いするとよいかをまとめたガイドである。

## 共通ルール

- まず既存資料を確認し、読めば分かることを質問しない。
- 1 回の質問は 1〜3 テーマに絞る。
- 抽象的な質問だけを投げず、回答例や選択肢を添える。
- 回答が難しそうなときは、仮説を置いて「近いかどうか」を確認する。
- ユーザーの負担が大きい場合は、今のフェーズに必要な最低限の確認から進める。

## 最初にお願いするとよい資料

- 企画書、要件定義、業務フロー、既存設計書
- 画面一覧、ワイヤーフレーム、API 仕様、DB スキーマ
- 運用手順、障害事例、監査要件、法規制
- KPI、成功指標、チーム体制、既存の用語集
- 既存のアクター一覧、ユースケース一覧、権限表、状態遷移図

## システムの目的整理で聞くこと

### 概要

- このシステムは誰のための何か。
- 新規開発か、既存業務の置き換えか、既存プロダクトの拡張か。
- いま 1 文で説明するならどう言うか。

### 解決する問題

- 現在どんな業務上の困りごとが起きているか。
- その問題を放置すると何が困るか。
- 問題が最も強く表れる場面や頻度はどこか。

### 対象ユーザー

- 主に使う人は誰か。現場担当者、管理者、外部顧客のどれに近いか。
- 利用者ごとに、成功と失敗をどう判断するか。
- 利用者以外に、承認者、監査担当、運用担当などの関係者はいるか。

### 目指す状態

- 導入後に何ができるようになっていれば成功か。
- 定量的に見たい指標はあるか。
- どこまで自動化、標準化、可視化したいか。

### 解決アプローチ

- どの業務判断をシステム化したいか。
- 人が残す判断と、システムへ委ねる判断をどう分けたいか。
- 外部システム、AI、手作業との役割分担はどうするか。

### プロダクトスコープ

- 最初のリリースで必ず含めたい範囲はどこまでか。
- あえて今回外すものは何か。
- 将来の拡張として想定している領域はあるか。

### 提供価値

- 誰に対して、どんな価値を提供できると考えているか。
- その価値は時間短縮、品質向上、売上、リスク低減のどれに近いか。
- 競合や既存運用と比べて、何が差別化ポイントか。

### 設計原則

- 業務で絶対に壊してはいけないルールは何か。
- 説明責任、監査性、追跡性、即時性のうち、特に重視するものはどれか。
- 今後の変更容易性で優先したい観点は何か。

## コアドメインとサブドメインを探るとき

### 軽量イベントストーミングの進め方

1. 主要なアクターを列挙する。
2. それぞれのアクターが、何をきっかけに何を行うかを時系列で並べる。
3. 行動の結果として何が起きたかを、過去形のイベントとして並べる。
4. そのイベントに影響するルール、例外、外部連携、承認ポイントを洗い出す。
5. 判断が難しい箇所、現場の痛みが強い箇所、差別化の核になる箇所をホットスポットとして残す。

### 質問例

- その業務は誰が始め、何が起きたら次の担当へ渡るか。
- 業務上「完了した」と判断する瞬間はいつか。
- 手戻り、差し戻し、キャンセルはどこで起こるか。
- 例外対応や属人化が起きやすいのはどこか。
- 他社や既存ツールと最も差がつく判断はどこか。

### 分類の補助

- コアドメイン
  - 競争優位の源泉で、ビジネス価値の中心になる領域。
- Supporting Subdomain
  - コアドメインを支えるが、差別化の主戦場ではない領域。
- Generic Subdomain
  - 汎用的で、既製品や共通基盤でも代替しやすい領域。

分類に迷うときは、次の観点で確認する。

- ここが弱いとプロダクト価値が大きく落ちるか。
- ここに独自ルールや高度な判断が集中しているか。
- 既製品で置き換えたときに競争力が失われるか。

用語のぶれが大きい場合は、早い段階で用語集テンプレートを追加し、設計書の正本をそろえる。

## 戦術的設計で聞くこと

### Value Object

- 同一性ではなく値で比較したい概念は何か。
- 生成時や更新時に必ず満たすべき条件は何か。
- 表示用の値と内部計算用の値を分ける必要はあるか。

### Entity

- 追跡すべき識別子は何か。
- どの状態遷移が業務上重要か。
- 履歴や監査が必要な変更はどれか。

### Aggregate

- どこまでを一貫性境界として同時に守る必要があるか。
- どの不変条件をトランザクション内で守りたいか。
- 同時更新の衝突が起きやすいのはどこか。

### Domain Service

- 1 つの Entity や Aggregate に閉じない業務判断はあるか。
- 複数概念をまたいで計算や判定を行う処理は何か。
- その処理は本当にドメイン知識か、アプリケーションの調停か。

### Repository

- どの単位で読み書きしたいか。
- 参照系と更新系で必要な検索条件は何か。
- 履歴、スナップショット、論理削除などの永続化要件はあるか。

### Application Service

- ユースケース開始時の入力は何か。
- 認可、トランザクション、外部連携の境界はどこか。
- 成功時と失敗時に呼び出し元へ何を返すか。

### Presentation

- 主な入口は Web、API、バッチ、管理画面のどれか。
- 利用者が入力しやすい表現と、ドメイン内部で扱いたい表現に差があるか。
- エラーや警告をどう伝えると現場で判断しやすいか。

## コンテキスト設計で聞くこと

- 何をきっかけにユースケースが開始されるか。
- 前提条件、事前チェック、権限確認は何か。
- 正常系の最短フローはどうなるか。
- 例外系、失敗時の戻し、再実行方針はどうするか。
- 保存対象、更新順序、同時実行制御はどうするか。
- 外部 API や通知基盤とのやりとりは同期か非同期か。
- ログ、監査証跡、メトリクスとして何を残すべきか。
- テストで最初に担保したい観点は何か。

## 情報提供を促す言い回し例

- 「既存の業務フローや画面があれば、そのまま共有してもらえると用語のずれを減らせます。」
- 「判断が難しい箇所だけでも教えてもらえると、コアドメインの見極めを進めやすいです。」
- 「正確な答えがなくても大丈夫なので、現場感に近いたたき台を教えて下さい。」
- 「用語が複数ある場合は、現場で最も自然に使う呼び方を優先して教えて下さい。」
