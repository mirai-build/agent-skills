# リポジトリ品質レビュー方法論

この Skill は、クリーンアーキテクチャ準拠かどうかだけで判断しない。
以下の 5 軸を共通評価軸とし、減点方式で総合評価を出す。

## 5 軸

1. 可読性
- 命名が意図を表しているか
- 関数やメソッドが長すぎないか
- ネストが深すぎないか
- コメントが理解の補助になっているか

2. 保守性
- 重複コードが目立たないか
- 修正時の影響範囲が局所化されているか
- 密結合や循環依存が強すぎないか

3. 設計品質
- 責務分離が適切か
- 依存関係が整理されているか
- ドメイン知識の配置が妥当か
- クリーンアーキテクチャの原則に大きく反していないか

4. テスト容易性
- 副作用が分離されているか
- 入出力が明確か
- 外部依存を差し替え可能か

5. 過剰設計の有無
- 不必要な抽象化がないか
- interface や層が過剰でないか
- 規模に対して複雑すぎないか

## 減点方式

- 初期点は 100 点とする。
- 問題点ごとに重要度を付けて減点する。
- 基本減点は `高=-15 / 中=-8 / 低=-3` とする。
- 同じ構造上の問題を複数軸へ重複計上しない。各指摘は主軸を 1 つに固定する。
- 些細な記法差やチーム規約の未確認だけで減点を積み上げすぎない。

## 重要度の基準

- 高
  - repo 横断または主要モジュールの変更容易性を大きく落とす問題
  - 構造崩壊、責務混線、強い密結合、依存方向の破綻、テスト不能に近い副作用集中など
- 中
  - 局所的だが再発しやすい設計・保守性問題
  - 長い関数、重複、feature 境界の曖昧さ、差し替えづらい外部依存など
- 低
  - 限定的な可読性や整理不足
  - 命名の弱さ、軽微な構成揺れ、補助コメント不足など

## ランク基準

- `A`: 85 点以上
- `B`: 70〜84 点
- `C`: 50〜69 点
- `D`: 49 点以下

ランクは「完成度」ではなく「保守しやすさの現時点評価」として扱う。

## 根拠の書き方

- 問題点には、可能な限り根拠ファイルやディレクトリを添える。
- 断定が難しい場合は「推測」「未確認」を明示する。
- 主観的な印象だけで終わらせず、「なぜ問題か」を変更容易性、理解容易性、再発可能性の観点で短く説明する。

## 規模別の見方

### 小規模 repo

- シンプルさを優先して評価する。
- 層や interface が少ないこと自体は減点しない。
- 逆に、規模に対して抽象化や分割が多すぎる場合は過剰設計として扱う。

### 中規模以上の repo

- ディレクトリ境界、依存方向、テスト配置、feature 単位の分割を重く見る。
- 変更波及を抑えられているか、責務が局所化されているかを重視する。

## 禁止事項

- クリーンアーキテクチャ準拠のみで高評価としない。
- 抽象化の多さをそのまま良さと判断しない。
- 主観だけで終わらせず、必ず理由を述べる。
