# 実装 Q&A 調査ガイド

この資料は、`mb-implementation-qa` が現状実装を読み解いて質問へ答えるときの正本である。
回答の主軸は常に現状コードとし、`docs/designs` は必要に応じた補助情報として照合する。

## 1. 基本姿勢

- 最初に質問の主論点を 1 つ決める。主論点が曖昧なまま広く探索しない。
- 入口、主要型、呼び出し元、類似実装、テストの順で、質問に近いものから読む。
- 直接の主張を支える根拠が 2〜4 ファイルでそろったら、周辺探索を止めて回答へ移る。
- `TODO`、仮実装、未使用コード、分岐だけある未接続処理は「存在するコード」ではあっても「実装済み機能」とは限らない。呼び出し元やテストの有無も合わせて確認する。

## 2. 実装方針の相談での読む順

1. entrypoint
2. interface / type
3. 類似実装
4. tests
5. 必要なら `docs/designs`

見る観点:

- 今回の責務は、どの layer や module に自然に属するか
- 既存実装に同じ境界の切り方、命名、依存方向があるか
- 新しい型や helper を足すべきか、それとも既存型を流用すべきか
- 類似ケースで例外処理、認可、永続化、副作用をどう扱っているか

回答の要点:

- 「どこへ置くべきか」と「なぜそこが自然か」をセットで返す。
- 既存パターンを踏襲するなら、どの実装を参照例とみなしたかを示す。
- 設計書がある場合は、既存コードの方針と整合しているか、未反映差分があるかも短く添える。

## 3. 実装内容の確認での読む順

1. call path
2. state change / side effect
3. persistence / API boundary
4. tests / TODO

見る観点:

- どの入口からどの処理が呼ばれるか
- 状態変更、副作用、外部呼び出しがどこで起きるか
- repository、ORM、HTTP client、queue、event publish などの境界をまたぐ地点はどこか
- テストが何を担保していて、どこが未検証か

回答の要点:

- 実装済みの範囲と、未実装または未接続の範囲を分けて述べる。
- 呼び出し経路を時系列で長く説明しすぎず、質問に関係する区間だけを要約する。
- 例外系、失敗時処理、再試行、監査が必要な処理では、その有無も確認して返す。

## 4. 差分確認の進め方

1. current code
2. design docs
3. mismatch summary

見る観点:

- 現状コードでは何が実装され、どの責務で動いているか
- 設計書では何が意図されているか
- 実装が先行しているのか、設計書が古いのか、実装漏れなのか

回答の要点:

- まず現状実装を `結論` に置く。
- 設計書との差分は `根拠` または `次の一手` で明示する。
- 差分が大きい場合でも、コードだけで断定できることと、設計書ベースで期待されることを混同しない。

## 5. 情報不足時の返し方

- 断定できないときは、憶測で埋めずに「何が足りないか」を先に示す。
- 次に確認すべきファイルや論点は 1〜3 件に絞る。
- 質問が広すぎる場合は、最も重要な主論点だけを先に回答し、残りは `次の一手` へ回す。
- 実装変更が必要なら `mb-ddd-backend-engineer`、設計更新が必要なら `mb-ddd-architect` を案内する。
